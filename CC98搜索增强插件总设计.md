# CC98 搜索增强插件总设计

## 1. 项目定位

本项目目标是做一个面向 CC98 的浏览器侧搜索增强插件。

插件不替代 CC98 官方搜索系统，不自建全站全文索引，不爬取全站内容，也不要求站方修改服务器。它的定位是：

> 在用户已有登录态和访问权限范围内，对 CC98 原生标题搜索做轻量增强。

核心思路是：

```text
用户输入搜索词
→ 插件解析查询
→ 生成若干增强搜索请求
→ 调用 CC98 原站搜索
→ 合并结果
→ 去重
→ 本地过滤
→ 本地排序
→ 展示类搜索引擎结果页
```

第一版只支持：

- 模糊搜索
- `+` 必须包含
- `-` 排除
- 多关键词搜索
- 按相关性、时间、热度综合排序

暂不支持复杂黑暗语法，例如 `title:`、`board:`、`author:`、`after:`、`before:`、`OR`、精确短语等。

## 2. 背景问题

CC98 原生搜索目前主要依赖标题搜索。实际使用中存在几个问题：

- 搜索词必须较接近标题原文。
- 中文分词效果不稳定。
- 简称、缩写、论坛黑话容易搜不到。
- 多关键词搜索的排序和召回效果有限。
- 用户很难判断为什么某些结果排在前面。

例如用户搜索：

```text
线性代数 作业
```

原站可能搜不到标题中写作：

```text
线代作业资料整理
```

插件要解决的不是完整全文搜索，而是提升标题搜索场景下的召回和排序体验。

## 3. 已确认的 CC98 搜索结构

通过前期探索，CC98 前端搜索入口和搜索结果结构如下。

### 3.1 顶部搜索框 DOM

稳定选择器：

```text
#searchText          搜索输入框
.searchIco           搜索按钮
.searchBoxSelect     当前搜索范围
.searchBoxSub li     搜索范围下拉项
```

首页搜索范围：

```text
主题 / 用户 / 版面
```

帖子页、版面页、搜索页中的搜索范围：

```text
版内 / 全站 / 用户 / 版面
```

回车逻辑：

```text
#searchText 按 Enter → 触发 .searchIco.click()
```

### 3.2 原站搜索路由

全站主题搜索：

```text
/search?boardId=0&keyword=<双重编码关键词>
```

版内搜索：

```text
/search?boardId=<boardId>&keyword=<双重编码关键词>
```

版面搜索：

```text
/searchBoard?keyword=<双重编码关键词>
```

### 3.3 API 结构

API base：

```text
https://api.cc98.org
```

主题搜索：

```text
GET /topic/search?keyword=<keyword>&size=20&from=<offset>
GET /topic/search/board/<boardId>?keyword=<keyword>&size=20&from=<offset>
```

版面搜索：

```text
GET /board/search?keyword=<keyword>
```

注意：

- 主题搜索 API 需要用户登录授权。
- 用户浏览器里登录，不等于插件后台裸请求自动有授权。
- 插件应优先在 CC98 页面上下文中请求或复用原搜索页结果，避免读取或外传 token。

### 3.4 搜索结果 DOM

搜索结果页结构：

```text
#focus-topic-area              搜索结果区域
.focus-topic-topicArea         结果列表容器
.focus-topic                   单条主题结果
.focus-topic-title             标题链接
.focus-topic-userName          作者
.focus-topic-board             所属版面
.focus-topic-info              时间、浏览、最后回复等信息
#focus-topic-getMore           加载更多
#noResult                      无结果提示
```

每次原站请求：

```text
size = 20
from = 当前已加载结果数
```

## 4. 第一版功能范围

### 4.1 模糊搜索

模糊搜索的目标是让用户不必完全输入标题原文。

第一版可以支持三类模糊：

1. 简单文本归一化

```text
大小写统一
全角半角统一
去除多余空格
常见符号归一化
```

2. 关键词宽松匹配

将用户输入拆成多个关键词，本地判断标题是否命中部分关键词。

例如：

```text
线性代数 作业
```

可以拆为：

```text
线性代数
作业
```

结果标题只要命中其中一部分，就可以参与排序。

3. 小型同义词/简称扩展

维护一个本地词典，例如：

```json
{
  "线性代数": ["线代", "linear algebra"],
  "数据结构": ["DS", "FDS"],
  "计算机系统": ["ICS", "CSAPP"],
  "微积分": ["微甲", "微乙", "calculus"]
}
```

第一版不做复杂分词和机器学习，只做可解释、可控的查询扩展。

### 4.2 多关键词搜索

用户输入：

```text
线代 期末 作业
```

插件将其解析为多个普通关键词：

```json
{
  "include": [],
  "exclude": [],
  "terms": ["线代", "期末", "作业"]
}
```

多关键词默认是弱 AND + 排序增强：

- 命中更多关键词的结果排名更高。
- 不要求所有关键词都必须命中。
- 若用户希望强制命中，应使用 `+`。

### 4.3 `+` 必须包含

语法：

```text
+线代 期末
```

含义：

- `线代` 是强制条件。
- 结果标题或可见摘要中必须命中 `线代` 或其高置信扩展词。
- `期末` 是普通关键词，只影响召回和排序。

解析结果：

```json
{
  "include": ["线代"],
  "exclude": [],
  "terms": ["期末"]
}
```

如果 `+线代` 扩展为：

```text
线代 / 线性代数 / linear algebra
```

则本地过滤时只要命中其中一个即可保留。

### 4.4 `-` 排除

语法：

```text
线代 -答案
```

含义：

- 搜索与 `线代` 相关的结果。
- 删除标题或可见摘要中包含 `答案` 的结果。

解析结果：

```json
{
  "include": [],
  "exclude": ["答案"],
  "terms": ["线代"]
}
```

排除词第一版不做主动扩展，避免误杀结果。

### 4.5 查询生成

插件根据用户输入生成有限数量的原站搜索 query。

示例：

```text
+线性代数 期末 -答案
```

解析：

```json
{
  "include": ["线性代数"],
  "exclude": ["答案"],
  "terms": ["期末"]
}
```

扩展：

```text
线性代数 → 线代 / linear algebra
期末 → final
```

生成搜索请求：

```text
线性代数 期末
线代 期末
linear algebra 期末
线性代数 final
```

限制：

```text
默认最多 4 个扩展 query
每个 query 默认只抓第一页
每页 20 条
默认串行或低并发请求
```

### 4.6 词库维护

第一版不应把课程简称、英文名和老师简称散落写在排序逻辑里。

实际实现采用独立词库文件：

```text
extension/lexicon.js
extension/teacher-lexicon.js
```

词库分三类：

```text
courseAliasGroups：课程名、简称、英文名
teacherAliasGroups：老师姓名、常用缩写
teacher-lexicon.js：从查老师详情页扫描生成的老师姓名、拼音首字母
segmentTerms：只用于切分无空格输入的普通词
shortQueryExpansions：单字中文查询的探索候选短语
```

同一个 alias group 内的词按强等效处理。例如：

```text
数字逻辑 / 数逻 / digital logic
微积分 / vif / 微甲 / 微乙
韩刚 / hg
```

后续扩充老师时，只需要追加：

```js
["老师姓名", "常用缩写"]
```

长期可以增加“候选词发现”能力：记录用户查询词、API 补召回命中的标题词、用户点击结果标题，再把高频共现词提交给人工确认，避免自动扩词导致误召回。

当前老师首字母词库通过扫描 `https://chalaoshi.de/t/{id}/` 的公开详情页生成：扫描 ID 范围 `1..12000`，有效详情页 `10295` 个，去重姓名 `9787` 个，生成姓名/首字母强等效组 `9786` 个，最大有效 ID `10933`。只保留姓名和首字母，不保留评分、评论、学院等详情信息。

单字中文查询不应进入强等效词库，而应走短词探索召回。比如 `梅` 不等于 `梅花`，但可以把 `梅花`、`梅雨`、`青梅` 当作补搜候选；排序阶段再对同一短语簇重复结果施加轻微降权，避免某个高频词簇占满第一页。

## 5. 结果页设计

插件应提供一个类似搜索引擎的结果页，而不是简单覆盖原列表。

每条结果至少展示：

```text
标题
所属版面
作者
发帖时间
最后回复时间
浏览数
回复数
命中的关键词
来源 query
综合分数或排序原因
```

推荐展示结构：

```text
[标题]
版面 · 作者 · 发帖时间 · 最后回复时间
命中：线代 ← 线性代数；期末
浏览 1234 · 回复 56
```

结果页还应展示搜索计划：

```text
已扩展搜索：
1. 线性代数 期末
2. 线代 期末
3. linear algebra 期末
4. 线性代数 final
```

这样用户可以理解结果从哪里来，也方便后续调试词典。

## 6. 排序模型

插件需要一个类似搜索引擎 PageRank 的本地排序模型。

这里不是真正的网页 PageRank，因为 CC98 搜索结果之间没有链接图结构。更准确地说，应设计一个本地 `SearchRank`：

```text
SearchRank = 相关性分 + 时间分 + 热度分
```

### 6.1 相关性分

相关性是主排序因素。

建议权重：

```text
原始 query 命中标题             +100
强制词 +term 命中               +80
普通关键词命中标题              +30 / 个
同义词高置信命中标题            +25 / 个
同义词低置信命中标题            +10 / 个
多个扩展 query 命中同一结果      +20 / 次
排除词命中                      直接删除
```

示例：

```text
用户搜索：线性代数 期末

结果 A：线性代数期末复习资料
结果 B：线代期末复习资料
结果 C：LA 复习资料

排序应倾向：
A > B > C
```

原因：

- A 命中原始词。
- B 命中高置信简称。
- C 命中低置信缩写，歧义较大。

### 6.2 时间分

时间分用于让较新的内容有一定优势，但不能压过强相关结果。

可基于最后回复时间或发帖时间计算。

推荐优先级：

```text
最后回复时间 lastPostTime
发帖时间 time
```

时间分示例：

```text
7 天内       +30
30 天内      +20
180 天内     +10
1 年内       +5
更早         +0
```

也可以使用连续衰减：

```text
timeScore = 30 * exp(-daysSinceLastPost / 90)
```

第一版建议先用分段模型，方便调试。

### 6.3 热度分

热度分体现浏览数和回复数。

可用字段：

```text
hitCount      浏览数
replyCount    回复数
```

推荐不要线性加分，否则老帖容易凭借历史浏览量碾压新帖。应使用对数压缩：

```text
hotScore = log10(hitCount + 1) * 8 + log10(replyCount + 1) * 15
```

回复数比浏览数更重要，因为回复通常代表更强互动。

### 6.4 综合排序公式

第一版推荐：

```text
finalScore =
  relevanceScore * 0.70
  + timeScore * 0.15
  + hotScore * 0.15
```

也可以提供三个排序模式：

```text
综合排序：相关性 70% + 时间 15% + 热度 15%
时间优先：相关性 40% + 时间 45% + 热度 15%
热度优先：相关性 40% + 时间 10% + 热度 50%
自定义：使用弹窗中的三个权重滑杆，并按总和归一化
```

默认使用综合排序。

## 7. 去重与合并

多个扩展 query 可能搜到同一个帖子。

去重 key 推荐：

```text
topicId
```

如果只能从 DOM 中拿 URL，则从 URL 中提取：

```text
/topic/<topicId>
```

合并规则：

```text
同一个 topicId 只保留一条结果
记录所有命中过它的 sourceQuery
记录所有命中的关键词和别名
命中 query 越多，相关性分越高
```

示例：

```json
{
  "topicId": 123,
  "title": "线代期末复习资料",
  "matchedQueries": ["线性代数 期末", "线代 期末"],
  "matchedTerms": ["线代", "期末"],
  "score": 168
}
```

## 8. 请求控制

为了避免给 CC98 服务器造成额外压力，第一版必须保守。

默认策略：

```text
只在用户按 Enter 或点击搜索时请求
不做输入时实时搜索
一次搜索最多 4 个扩展 query
每个 query 默认只取第一页
请求并发最多 1 或 2
相同 query 缓存 10 分钟
失败后不连续重试
```

可选深搜按钮：

```text
扩展 query 最多 8 个
每个 query 取前 2 页
用户主动点击后才执行
```

## 9. 推荐实现架构

### 9.1 Content Script

负责：

```text
发现 CC98 搜索框
接管搜索提交
渲染增强结果页
读取当前页面上下文中的 boardId
在用户登录态范围内发起搜索
```

关键点：

```text
不要把用户认证信息传到外部服务器
不要读取不必要的 cookie/localStorage
优先使用页面已有权限进行请求
```

### 9.2 Background / Service Worker

负责：

```text
保存配置
保存同义词词典
保存搜索缓存
处理扩展设置页
```

如果后台无法直接带上 CC98 登录态，则不要让后台直接裸请求主题搜索 API。

### 9.3 搜索核心模块

独立模块：

```text
parseQuery(input)
expandQuery(parsedQuery, aliasDict)
runOriginalSearch(searchPlan)
mergeResults(results)
filterResults(results, parsedQuery)
rankResults(results, rankingMode)
```

这样后续可以单独测试解析、扩展、排序，不依赖浏览器页面。

## 10. MVP 验收标准

第一版完成后，应能做到：

1. 用户在 CC98 顶部搜索框输入：

```text
线性代数 期末
```

插件能扩展出：

```text
线性代数 期末
线代 期末
linear algebra 期末
```

2. 用户输入：

```text
+线代 -答案 期末
```

插件能：

```text
保留必须包含线代相关词的结果
过滤标题中包含答案的结果
根据期末命中情况加分
```

3. 结果页能展示：

```text
标题
版面
作者
时间
热度
命中解释
排序模式
```

4. 排序能体现：

```text
更相关的结果排前面
较新的结果有适度加权
高回复/高浏览结果有适度加权
```

5. 请求控制符合：

```text
默认最多 4 个扩展 query
不实时搜索
有缓存
不绕权限
不外传认证信息
```

## 11. 可行性评估

该方案整体可行。

原因：

- CC98 搜索入口 DOM 稳定，适合 content script 接管。
- 原站已经有明确搜索页和搜索 API。
- 搜索结果字段足够做本地重排，包括标题、版面、作者、时间、浏览、回复。
- 第一版只做轻量增强，不涉及全文索引、数据库同步、ES 运维。
- 查询扩展和本地排序都可以在浏览器内完成。

主要风险：

- 主题搜索 API 需要登录授权，后台裸请求不可行。
- React 前端 class 名目前可用，但未来站点改版可能变化。
- 模糊扩展词典质量会直接影响结果质量。
- 扩展 query 太多会增加服务器压力。
- 只基于标题搜索，无法解决正文命中问题。

风险控制：

- 优先在页面上下文中运行搜索。
- 保守限制请求数量。
- 搜索计划透明展示。
- 排序原因可解释。
- 词典分高低置信度。
- 保留关闭插件增强的选项。

## 12. 当前结论

第一版不应该做复杂搜索引擎，而应该做：

> CC98 标题搜索的本地增强层。

最小可行版本是：

```text
模糊词典扩展
+ 必须包含
- 排除
多关键词
多 query 原站搜索
合并去重
SearchRank 综合排序
类搜索引擎结果页
```

这个范围足够解决 CC98 当前搜索里最痛的简称、缩写、多关键词召回问题，同时不会掉进服务端全文搜索系统的工程泥潭。
